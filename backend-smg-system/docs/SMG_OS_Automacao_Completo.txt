1. Visão Geral
O módulo de automação tem um único objetivo: converter leads em diagnósticos agendados, sem
intervenção humana. Todo o fluxo conversacional é conduzido por IA via WhatsApp. O sistema opera com
dois tipos de lead — outbound (prospecção ativa) e inbound (chegada por iniciativa própria) — e dois
workflows independentes.

4. Etapas do Fluxo Conversacional
Etapas 1 a 5 — Exclusivas do fluxo outbound
 1
Abordagem Inicial — Outbound
Lead entra com status NOVO_LEAD. IA envia primeira mensagem apresentando a SMG. Tom consultivo,
sem pitch. Personalizada por segmento usando template de config_templates (campo etapa + segmento).
Status: NOVO_LEAD — aguardando resposta
Banco: leads_automacao_timeline: INSERT tipo "mensagem_enviada". UPDATE ultima_interacao.
 2
Qualificação Inicial — Outbound
Lead responde. IA classifica intenção: interesse / objeção / dúvida / sem contexto. Com interesse mínimo, IA
faz perguntas para entender o contexto operacional. Objetivo: confirmar fit básico antes de avançar.
Status: Mantém NOVO_LEAD. Se sem resposta: inicia sequência FUP.
Banco: leads_automacao_timeline: INSERT tipo "mensagem_recebida". UPDATE ultima_interacao = now().
 3
Identificação de Decisor — Outbound
IA pergunta se o lead é o responsável pela decisão. Inserida naturalmente na conversa. Dois caminhos:
decisor confirmado (continua) ou intermediário (ver Seção 5).
Status: INTERMEDIARIO_IDENTIFICADO ou continua fluxo normal
Banco: UPDATE leads_automacao SET status = "INTERMEDIARIO_IDENTIFICADO" quando intermediário
confirmado.
 4
Preparação para Formulário — Outbound
IA apresenta o formulário como próximo passo natural. Enquadra como ferramenta para gerar análise
personalizada gratuita. Deve parecer benefício, não pesquisa. Envia link: smg.com.br/diagnostico.
Status: Mantém status atual — aguardando preenchimento
Banco: leads_automacao_timeline: INSERT tipo "link_formulario_enviado".
 5
Formulário Respondido — Outbound
Lead preenche o formulário no site. O frontend cria registro em leads_diagnostico com token e atualiza
leads_automacao.status = FORMULARIO_ENVIADO. N8N detecta via Realtime ou polling e avança para
FORMULARIO_RESPONDIDO.
Status: FORMULARIO_ENVIADO → FORMULARIO_RESPONDIDO
Banco: UPDATE status = "FORMULARIO_RESPONDIDO". Ler dados de leads_diagnostico via
diagnostico_formulario_id.
Etapas 6 a 8 — Comuns a outbound e inbound
 6
Geração e Entrega da Análise de Maturidade
N8N lê dados de leads_diagnostico via diagnostico_formulario_id. Monta prompt com os dados e o
framework SIG. LLM gera a análise completa (avaliação por pilar: Sistema / Inteligência / Gestão, gargalos,
pré-diagnóstico). N8N gera PDF e envia via WhatsApp como documento. Para inbound: esta é a primeira
mensagem recebida pelo lead — deve ser direta e de alto valor.
Status: FORMULARIO_RESPONDIDO → ANALISE_ENVIADA
Banco: UPDATE status = "ANALISE_ENVIADA". INSERT em analises_maturidade.
leads_automacao_timeline: INSERT "analise_enviada".

Página 7
 7
Conversão para Diagnóstico
IA retoma a conversa após a análise. Para inbound: referenciar dados do formulário na abertura —
demonstrar personalização, não fazer apresentação genérica. Para outbound: continuar o fio da conversa já
estabelecida. IA valida fit. Se não houver fit: DESQUALIFICADO com automation_active = false.
Status: ANALISE_ENVIADA → DECISOR_IDENTIFICADO ou DESQUALIFICADO
Banco: UPDATE status conforme decisão. Se DESQUALIFICADO: automation_active = false.
 8
Agendamento do Diagnóstico
IA conduz a conversa até definição de data e hora via mensagem. Captura e registra. Quando confirmado:
INSERT em leads_crm (idempotente), automation_active = false, notificação ao comercial conforme
config_notificacoes.
Status: DECISOR_IDENTIFICADO → DIAGNOSTICO_AGENDADO
Banco: UPDATE status = "DIAGNOSTICO_AGENDADO", automation_active = false. INSERT leads_crm
com lead_automacao_id, horario_reuniao, data_prevista. Disparar config_notificacoes.
Arquitetura assíncrona: o N8N não aguarda resposta em loop. Cada mensagem recebida é um evento
independente que lê o estado atual do lead do banco e decide o próximo passo.

5. Fluxo de Bifurcação — Intermediário vs Decisor
5.1 Caminho: Decisor confirmado
 IA confirma que o lead é o responsável pela decisão
 Fluxo continua normalmente pelas etapas 4 a 8 (outbound) ou 6 a 8 (inbound)
5.2 Caminho: Intermediário identificado
 UPDATE leads_automacao SET status = "INTERMEDIARIO_IDENTIFICADO", automation_active =
false no lead original
 IA solicita o número de WhatsApp do decisor
 Quando obtido: validar formato E.164, verificar duplicata em leads_automacao
 INSERT novo lead: telefone do decisor, status = "NOVO_LEAD", mesmo segmento e empresa
 WF2 reinicia do zero com o novo lead (decisor)
Aplicar a mesma validação E.164 ao número do decisor recebido via mensagem. Números enviados
pelo WhatsApp podem vir em formatos variados — normalizar antes do INSERT.

6. Fluxo Inbound — Detalhamento
Lead inbound = lead que chegou por iniciativa própria e já preencheu o formulário. Possui intenção
declarada. Deve ser tratado como lead quente desde o primeiro contato. Entrar no WF2 como
NOVO_LEAD é erro crítico.
6.1 Origens de leads inbound
 Formulário no site (smg.com.br/diagnostico)
 Instagram — link do formulário compartilhado via DM ou bio
 Outros canais de entrada direta futuros
6.2 Como o frontend registra o lead inbound
DiagnosticoPage.tsx já implementa o fluxo:
 INSERT em leads_diagnostico com token único gerado pelo banco
 INSERT em leads_automacao com status = FORMULARIO_ENVIADO, pipeline_origin =
"diagnostico_site", canal_aquisicao = "inbound_site"
 Página de sucesso monta link WhatsApp com "SMG-{token}" na mensagem
O frontend cria o lead como FORMULARIO_ENVIADO (não FORMULARIO_RESPONDIDO). O N8N
avança para FORMULARIO_RESPONDIDO ao receber a mensagem WhatsApp com o token,
confirmando que o contato existe e está ativo.
6.3 Identificação via token
 Lead envia mensagem WhatsApp começando com "SMG-{token}"
 N8N extrai o token, busca em leads_diagnostico WHERE token = {token}
 Vincula ao lead em leads_automacao via diagnostico_formulario_id
 UPDATE status = "FORMULARIO_RESPONDIDO" — WF2 detecta e inicia Etapa 6

7. Follow-up e Janela do WhatsApp
7.1 Regra da janela de 24h
A API do WhatsApp permite mensagens livres apenas dentro de 24h após a última mensagem do lead.
Fora da janela: somente Templates aprovados pela Meta.
O N8N rastreia ultima_interacao em leads_automacao. Toda mensagem recebida: UPDATE
ultima_interacao = now(). Antes de cada envio: verificar se now() - ultima_interacao < 24h.

Toda configuração de horário vive no banco (config_automacao). O N8N não deve ter lógica de horário

9.2 Lógica de verificação antes de cada envio
 SELECT * FROM config_automacao LIMIT 1
 Verificar se o dia atual está em dias_personalizados
 Verificar se horário atual está entre horario_inicio e horario_fim (fuso: America/Sao_Paulo)
 Se fora do horário: enfileirar para o próximo horário válido — nunca descartar
 Aplicar delay aleatório entre tempo_resposta_min e tempo_resposta_max
Todos os timestamps do banco estão em UTC. Converter para America/Sao_Paulo ao comparar com os
horários configurados.

13. Arquitetura para Escalabilidade
Centenas de leads simultâneos no curto prazo, milhares no médio prazo. Arquitetura assíncrona
obrigatória desde o início.
13.1 Princípios obrigatórios
 Stateless por execução: cada execução lê o estado do lead do banco. Sem memória de execuções
anteriores.
 Processamento independente por lead: um lead nunca bloqueia outro.
 Idempotência: toda operação é segura para reexecução. Verificar existência antes de inserir.
 Fila de envio: mensagens enfileiradas com delay entre mensagens do mesmo lead.
 Rate limiting: controlar taxa respeitando limites da API Meta (~80 msg/seg em contas Tier 1).
